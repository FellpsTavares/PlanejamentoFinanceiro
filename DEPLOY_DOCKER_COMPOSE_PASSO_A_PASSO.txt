DEPLOY COM DOCKER COMPOSE - ELO FINANCEIRO
Data: 13/03/2026

OBJETIVO
- Subir o sistema em servidor usando Docker Compose.
- Persistir dados no host em: /media/devmon/DATA/dados-felps
- Facilitar atualização/rollback sem reinstalar dependências manualmente.
- Operar com dois modos: local (teste) e produção (domínio real).

--------------------------------------------------
1) COMO O DOCKER VAI AGIR NESTE PROJETO
--------------------------------------------------
1. O Docker cria containers isolados para cada serviço:
   - db: PostgreSQL
   - backend: Django + Gunicorn
   - web: Nginx + frontend React (build estático)

2. O Docker Compose coordena esses containers:
   - sobe os serviços na mesma rede interna
   - permite comando único para subir/parar/recriar
   - garante ordem básica de inicialização (web depende do backend)

3. No fluxo de requisições:
   - usuário acessa porta 80 (serviço web/Nginx)
   - Nginx entrega frontend estático (React)
   - chamadas /api, /admin e /healthz são encaminhadas para o backend

4. Persistência:
   - tudo que está em volume bind mount não é perdido ao recriar container
   - path do host foi definido em /media/devmon/DATA/dados-felps

3) VOLUMES USADOS (PERSISTÊNCIA)
--------------------------------------------------
Banco PostgreSQL (escrita):
- /media/devmon/DATA/dados-felps/postgres/data:/var/lib/postgresql/data

Backend (escrita):
- /media/devmon/DATA/dados-felps/backend/media:/app/media
- /media/devmon/DATA/dados-felps/backend/staticfiles:/app/staticfiles

Web/Nginx (somente leitura):
- /media/devmon/DATA/dados-felps/backend/media:/srv/media:ro
- /media/devmon/DATA/dados-felps/backend/staticfiles:/srv/static:ro

Explicação:
- db grava dados do PostgreSQL em /var/lib/postgresql/data
- backend grava uploads em /app/media e arquivos estáticos coletados em /app/staticfiles
- Nginx lê esses mesmos diretórios no host e publica em /media e /static

--------------------------------------------------
4) PRÉ-REQUISITOS NO SERVIDOR
--------------------------------------------------
1. Instalar Docker e Docker Compose plugin.
2. Garantir que a aplicação está no servidor (git clone ou upload).
3. Criar arquivo de ambiente de produção:
   - copiar fintech-saas/.env.docker.production.example para fintech-saas/.env.docker.production
   - ajustar credenciais e chaves

4. (Opcional) Para testes locais com Compose:
   - copiar fintech-saas/.env.docker.local.example para fintech-saas/.env.docker.local

Variáveis obrigatórias no modo produção:
   - SECRET_KEY (forte e aleatória)
   - DEBUG=False
   - APP_ENV=production
   - ALLOWED_HOSTS=elofinanceiro.com.br,www.elofinanceiro.com.br,seu-ip
   - CORS_ALLOWED_ORIGINS=https://elofinanceiro.com.br,https://www.elofinanceiro.com.br
   - CSRF_TRUSTED_ORIGINS=https://elofinanceiro.com.br,https://www.elofinanceiro.com.br
   - POSTGRES_DB=elofinanceiro
   - POSTGRES_USER=elofinanceiro
   - POSTGRES_PASSWORD=troque_esta_senha_forte
   - DB_HOST=db
   - DB_PORT=5432
   - DB_SSLMODE=disable

Como gerar SECRET_KEY (no servidor ou local):
   python -c "import secrets; print(secrets.token_urlsafe(64))"

Observação importante:
- O projeto está configurado para PostgreSQL (não SQLite) em produção.

--------------------------------------------------
5) PRIMEIRA SUBIDA (PASSO A PASSO)
--------------------------------------------------
No servidor, dentro da pasta raiz do projeto (onde está docker-compose.yml):

1) Criar pastas de persistência
   sudo mkdir -p /media/devmon/DATA/dados-felps/postgres/data
   sudo mkdir -p /media/devmon/DATA/dados-felps/backend/media
   sudo mkdir -p /media/devmon/DATA/dados-felps/backend/staticfiles
   sudo mkdir -p /media/devmon/DATA/dados-felps/letsencrypt/conf
   sudo mkdir -p /media/devmon/DATA/dados-felps/letsencrypt/www

2) Ajustar permissão para evitar erro de escrita
   sudo chown -R 999:999 /media/devmon/DATA/dados-felps/postgres
   sudo chown -R 1000:1000 /media/devmon/DATA/dados-felps/backend

3) Build e subida em background (produção)
   docker compose -f docker-compose.yml -f docker-compose.production.yml up -d --build

4) Emitir certificado TLS (primeira vez)
   docker compose -f docker-compose.yml -f docker-compose.production.yml run --rm certbot certonly --webroot -w /var/www/certbot -d elofinanceiro.com.br -d www.elofinanceiro.com.br --email SEU_EMAIL --agree-tos --no-eff-email

5) Reiniciar web para carregar configuração HTTPS
   docker compose -f docker-compose.yml -f docker-compose.production.yml up -d web

6) Build e subida local para testes (porta 8080)
   docker compose -f docker-compose.yml -f docker-compose.local.yml up -d --build

7) Acompanhar logs
   docker compose -f docker-compose.yml -f docker-compose.production.yml logs -f backend
   docker compose -f docker-compose.yml -f docker-compose.production.yml logs -f web
   docker compose -f docker-compose.yml -f docker-compose.production.yml logs -f db

8) Validar saúde
   curl http://SEU_IP/healthz
   Esperado: {"status":"ok"}

8.1) Validar banco em execução
   docker compose -f docker-compose.yml -f docker-compose.production.yml exec db pg_isready -U elofinanceiro -d elofinanceiro

9) Validar domínio
   - Aponte DNS tipo A de elofinanceiro.com.br para o IP público do servidor
   - (Opcional) Aponte www com CNAME para elofinanceiro.com.br
   - Teste: http://elofinanceiro.com.br/healthz
   - Teste HTTPS: https://elofinanceiro.com.br/healthz

9.1) Registro.br (DNS) - preenchimento recomendado
    - Clique em "Configurar endereçamento" -> "Modo avançado"
    - Registro A:
       Nome: @ (ou vazio)
       Valor: IP público do servidor
    - Registro CNAME:
       Nome: www
       Valor: elofinanceiro.com.br.
    - TTL: padrão (300 ou 3600)
    - Não usar "Alterar servidores DNS" se já estiver com DNS do Registro.br

7) ATUALIZAÇÃO EM PRODUÇÃO (FLUXO SEGURO)
--------------------------------------------------
1. Atualizar código (git pull).
2. Conferir variáveis em fintech-saas/.env.docker.production.
3. Recriar imagens e subir:
   docker compose -f docker-compose.yml -f docker-compose.production.yml up -d --build
4. Ver logs e healthcheck.
5. Testar login e endpoint autenticado.
6. Renovar certificado manualmente (quando necessário):
   docker compose -f docker-compose.yml -f docker-compose.production.yml run --rm certbot renew --webroot -w /var/www/certbot

--------------------------------------------------
8) TROUBLESHOOTING RÁPIDO
--------------------------------------------------
Erro de CORS/CSRF:
- revisar CORS_ALLOWED_ORIGINS e CSRF_TRUSTED_ORIGINS no fintech-saas/.env.docker.production
- garantir domínio com https correto

Erro 400/DisallowedHost:
- revisar ALLOWED_HOSTS

Erro backend não conecta no banco:
- revisar DB_HOST=db, DB_PORT=5432 e credenciais iguais às do serviço db no compose

Erro de redirect infinito no teste local (/healthz 301):
- no modo produção isso é esperado com SECURE_SSL_REDIRECT=true sem HTTPS ativo
- para teste local, usar modo local (docker-compose.local.yml) que desativa redirect SSL

--------------------------------------------------
9) COMO GARANTIR QUE ESTÁ PRONTO PARA PRODUÇÃO
--------------------------------------------------
Checklist mínimo antes do go-live:
1. docker compose -f docker-compose.yml -f docker-compose.production.yml ps -> db (healthy), backend (up), web (up)
2. docker compose -f docker-compose.yml -f docker-compose.production.yml exec backend python manage.py migrate --noinput -> sem erro
3. docker compose -f docker-compose.yml -f docker-compose.production.yml exec backend python manage.py check --deploy -> sem erro crítico
4. Teste de health:
   - interno: docker compose -f docker-compose.yml -f docker-compose.production.yml exec backend python -c "import urllib.request; req=urllib.request.Request('http://127.0.0.1:8000/healthz/', headers={'X-Forwarded-Proto':'https'}); print(urllib.request.urlopen(req).read().decode())"
   - externo: https://elofinanceiro.com.br/healthz
5. Teste funcional manual:
   - login
   - criação de transação
   - listagem de transações
   - acesso a /admin
6. Teste de persistência:
   - reiniciar stack: docker compose -f docker-compose.yml -f docker-compose.production.yml down && docker compose -f docker-compose.yml -f docker-compose.production.yml up -d
   - validar se dados continuam presentes
7. Backup:
   - confirmar rotina de backup de /media/devmon/DATA/dados-felps
   - confirmar backup do banco PostgreSQL

Automatização do checklist (recomendado):
- Script: scripts/pre_go_live_compose.sh
- Executar no servidor Linux:
   chmod +x scripts/pre_go_live_compose.sh
   ./scripts/pre_go_live_compose.sh

Se DNS/SSL ainda estiver em propagação:
- ./scripts/pre_go_live_compose.sh --skip-external

--------------------------------------------------
9.1) CENÁRIO SEM ACESSO AO SERVIDOR (Handoff)
--------------------------------------------------
Se somente o dono do servidor tem acesso, enviar para ele:
1. Subir stack:
   docker compose -f docker-compose.yml -f docker-compose.production.yml up -d --build
2. Validar serviços:
   docker compose -f docker-compose.yml -f docker-compose.production.yml ps
3. Validar banco:
   docker compose -f docker-compose.yml -f docker-compose.production.yml exec db pg_isready -U elofinanceiro -d elofinanceiro
4. Rodar migrations:
   docker compose -f docker-compose.yml -f docker-compose.production.yml exec backend python manage.py migrate --noinput
5. Emitir SSL (após DNS ok):
   docker compose -f docker-compose.yml -f docker-compose.production.yml run --rm certbot certonly --webroot -w /var/www/certbot -d elofinanceiro.com.br -d www.elofinanceiro.com.br --email SEU_EMAIL --agree-tos --no-eff-email
6. Recarregar web:
   docker compose -f docker-compose.yml -f docker-compose.production.yml up -d web

Erro de escrita em media/static:
- revisar permissões no path /media/devmon/DATA/dados-felps/backend
