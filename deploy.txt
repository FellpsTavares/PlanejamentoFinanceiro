LEVANTAMENTO DE DEPLOY - ELO FINANCEIRO (VERCEL + RENDER + BANCO GERENCIADO)
Data: 11/03/2026

==================================================
1) DIAGNOSTICO ATUAL
==================================================
- Backend Django ainda preparado para SQLite por padrao (nao ideal para producao).
- Frontend React/Vite usa URL de API local fixa (localhost), o que impede acesso externo em producao.
- Configuracao atual focada em desenvolvimento local.

Conclusao:
Para colocar o sistema em producao real para clientes de diversas regioes do Brasil, e necessario:
1) Migrar para PostgreSQL gerenciado.
2) Configurar variaveis de ambiente de producao.
3) Ajustar CORS/CSRF/hosts.
4) Publicar frontend no Vercel e backend no Render.

==================================================
2) ARQUITETURA RECOMENDADA (GRATUITA PARA INICIO)
==================================================
Frontend:
- Vercel (site React com CDN global).

Backend:
- Render Web Service (Django + Gunicorn).

Banco de dados (sugestoes):
1) Neon (PostgreSQL) - recomendado para comecar; simples, bom free tier.
2) Supabase (PostgreSQL) - recomendado; painel e ecossistema muito bons.
3) Render Postgres - pode ser opcao, mas conferir disponibilidade/limites no plano free atual.

Recomendacao pratica:
- Para iniciar: Vercel + Render + Neon.

==================================================
3) ALTERACOES NECESSARIAS NO PROJETO
==================================================
Backend (Django):
- Trocar SQLite por DATABASE_URL (PostgreSQL).
- Adicionar dependencias de producao:
  - gunicorn
  - dj-database-url
  - psycopg2-binary
  - whitenoise
- Definir DEBUG=false em producao.
- Configurar ALLOWED_HOSTS com dominio do Render e dominio final.
- Configurar CORS_ALLOWED_ORIGINS com dominio do frontend Vercel.
- Configurar CSRF_TRUSTED_ORIGINS com dominios HTTPS.
- Configurar SECURE_PROXY_SSL_HEADER e cookies seguros em producao.
- Manter SECRET_KEY forte via variavel de ambiente.

Frontend (Vite/React):
- Substituir URL fixa da API por variavel de ambiente:
  - VITE_API_URL=https://SEU_BACKEND_RENDER/api
- Garantir build com npm run build.
- Criar configuracao de rewrite SPA (vercel.json) para evitar 404 em refresh de rota.

Operacao:
- Criar endpoint de health check (/healthz) no backend (recomendado).
- Rodar migrate e collectstatic no deploy do backend.
- Validar fluxo completo: login, cadastro, transacoes, relatorios e permissao de superusuario.

==================================================
4) PASSO A PASSO - DEPLOY REAL
==================================================
PASSO 0 - PREPARACAO
1. Garantir codigo atualizado no GitHub (backend e frontend).
2. Criar branch principal estavel para deploy.

PASSO 1 - BANCO DE DADOS (NEON OU SUPABASE)
1. Criar conta no Neon (ou Supabase).
2. Criar projeto Postgres.
3. Copiar a string de conexao DATABASE_URL.
4. Guardar essa URL para usar no Render.

PASSO 2 - BACKEND NO RENDER
1. Acessar Render -> New -> Web Service.
2. Conectar repositorio GitHub.
3. Selecionar a pasta/projeto do backend.
4. Configurar:
   - Build Command:
     pip install -r requirements.txt && python manage.py collectstatic --noinput && python manage.py migrate
   - Start Command:
     gunicorn config.wsgi:application
5. Definir variaveis de ambiente no Render:
   - SECRET_KEY
   - DEBUG=false
   - DATABASE_URL
   - ALLOWED_HOSTS
   - CORS_ALLOWED_ORIGINS
   - CSRF_TRUSTED_ORIGINS
   - ASSISTANT_CHAT_ENABLED=false (se mantiver chat desativado)
6. Fazer deploy e validar endpoint basico da API.

PASSO 3 - FRONTEND NO VERCEL
1. Acessar Vercel -> Add New Project.
2. Conectar mesmo repositorio GitHub.
3. Definir Root Directory do frontend.
4. Configurar:
   - Build command: npm run build
   - Output directory: dist
5. Adicionar env var no Vercel:
   - VITE_API_URL=https://SEU_BACKEND_RENDER/api
6. Deploy e obter URL publica do frontend.

PASSO 4 - AJUSTES CRUZADOS
1. Voltar no Render e adicionar o dominio do Vercel em CORS_ALLOWED_ORIGINS e CSRF_TRUSTED_ORIGINS.
2. Garantir ALLOWED_HOSTS contem dominio do Render e (se necessario) dominio custom.
3. Fazer redeploy backend.

PASSO 5 - VALIDACAO FINAL (SMOKE TEST)
1. Abrir sistema pela URL do Vercel.
2. Testar login.
3. Testar cadastro de conta.
4. Testar criacao/listagem de transacoes.
5. Testar modulos habilitados (transport/investments).
6. Testar permissao de superusuario.
7. Testar comportamento com cliente em rede externa (4G/outra internet).

==================================================
5) ESTIMATIVA DE TEMPO
==================================================
Cenario normal (sem bloqueios):
- Ajustes de codigo/config: 2h a 4h
- Deploy backend + banco: 1h30 a 3h
- Deploy frontend + integracao: 1h a 2h
- Testes e correcoes finais: 2h a 4h

Total estimado:
- 6h30 a 13h (aprox. 1 dia util)

Pode aumentar se houver:
- problema de CORS/CSRF,
- migracoes com dados inconsistentes,
- necessidade de limpeza de base,
- configuracao de dominio proprio e DNS (propagacao pode levar horas).

==================================================
6) RISCOS E LIMITACOES DO PLANO GRATUITO
==================================================
- Cold start no backend (latencia no primeiro acesso apos inatividade).
- Limites de CPU/memoria e horas de uso.
- Ausencia de SLA para ambiente critico.

Recomendacao para producao com clientes pagantes:
- Evoluir para plano pago minimo no backend e banco assim que houver uso real continuo.
